# Swagger API Documentation Setup

This document explains how we set up Swagger documentation for the Vive Real Estate API.

## Installation

First, we installed the required packages:

```bash
npm install @nestjs/swagger swagger-ui-express
```

## Configuration

### 1. Main Application Setup

In `main.ts`, we configured Swagger with the following settings:

```typescript
// Swagger configuration
const config = new DocumentBuilder()
  .setTitle('Vive Real Estate API')
  .setDescription('API documentation for Vive Real Estate application')
  .setVersion('1.0')
  .addBearerAuth()
  .build();

const document = SwaggerModule.createDocument(app, config);
SwaggerModule.setup('api', app, document);
```

This setup:
- Sets the API title and description
- Configures version information
- Adds Bearer token authentication support
- Makes the Swagger UI available at the `/api` endpoint

### 2. DTO Documentation

We created several DTOs with Swagger decorators to document our API requests and responses:

#### Create Payment DTO
```typescript
export class CreatePaymentDto {
  @ApiProperty({
    description: 'The amount to be paid',
    example: 1500.00,
    minimum: 0,
  })
  @IsNumber()
  @IsPositive()
  amount: number;

  @ApiProperty({
    description: 'The payment method to be used',
    enum: PaymentMethod,
    example: PaymentMethod.BANK_TRANSFER,
  })
  @IsEnum(PaymentMethod)
  paymentMethod: PaymentMethod;
}
```

#### Response DTOs
We created three response DTOs to document our API responses:

1. `PaymentInfoResponseDto`: Documents the payment information endpoint response
2. `PaymentHistoryResponseDto`: Documents the payment history endpoint response
3. `CreatePaymentResponseDto`: Documents the create payment endpoint response

### 3. Controller Documentation

In the `PaymentsController`, we added Swagger decorators to document our endpoints:

```typescript
@ApiTags('Tenant Payments')
@Controller('tenant')
@UseGuards(JwtAuthGuard)
@ApiBearerAuth()
export class PaymentsController {
  @Get('payment-info')
  @ApiOperation({ summary: 'Get tenant payment information' })
  @ApiResponse({
    status: 200,
    description: 'Returns the current amount due and property information',
    type: PaymentInfoResponseDto,
  })
  // ... additional response decorators
}
```

## Available Endpoints

The following endpoints are documented in Swagger:

### 1. Get Payment Information
- **URL**: `GET /tenant/payment-info`
- **Description**: Returns the current amount due and property information
- **Authentication**: Required (Bearer Token)
- **Response**: `PaymentInfoResponseDto`

### 2. Create Payment
- **URL**: `POST /tenant/payments`
- **Description**: Creates a new payment
- **Authentication**: Required (Bearer Token)
- **Request Body**: `CreatePaymentDto`
- **Response**: `CreatePaymentResponseDto`

### 3. Get Payment History
- **URL**: `GET /tenant/payment-history`
- **Description**: Returns the list of all payments made by the tenant
- **Authentication**: Required (Bearer Token)
- **Response**: `PaymentHistoryResponseDto`

## Accessing the Documentation

1. Start your backend server
2. Navigate to `http://localhost:3001/api`
3. You'll see the Swagger UI with all documented endpoints
4. Use the "Authorize" button to add your Bearer token for testing authenticated endpoints

## Features

The Swagger documentation includes:

1. **Interactive Testing**
   - Try out API endpoints directly from the UI
   - View request/response schemas
   - Test with authentication

2. **Detailed Documentation**
   - HTTP methods
   - Request parameters
   - Response types
   - Status codes
   - Authentication requirements

3. **Example Values**
   - Sample request payloads
   - Sample response formats
   - Enum values

4. **Error Responses**
   - 401 Unauthorized
   - 403 Forbidden
   - 404 Not Found
   - 400 Bad Request

## Best Practices

1. **Keep Documentation Up-to-Date**
   - Update DTOs and decorators when making API changes
   - Add new response types for new endpoints
   - Document all possible error scenarios

2. **Use Clear Descriptions**
   - Provide meaningful descriptions for all properties
   - Include examples where helpful
   - Document validation rules

3. **Group Related Endpoints**
   - Use `@ApiTags` to organize endpoints
   - Keep related endpoints together
   - Use consistent naming conventions

4. **Document Authentication**
   - Clearly indicate which endpoints require authentication
   - Document the authentication method
   - Provide examples of authentication headers
